# URL Reducing Service

В данном репозитории реализован сервис для генерации коротких ссылок. Проект реализован на Go 1.25, работает в двух режимах (с хранением данных в памяти приложения или в PostgreSQL), использует Docker Compose и следует принципам чистой архитектуры (Clean Architecture).

## Запуск приложения

### 1. Запуск через Docker Compose (решение для продакшена на PostgreSQL)

Для удобства проверки сервис полностью обернут в Docker Compose.

Для запуска в терминале выполните: 
```bash
docker compose up --build -d
```
После успешной сборки и инициализации сервис будет доступен на: `http://localhost:8080`. 
База данных поднимется автоматически, SQL-миграции так же применятся автоматически.

Для остановки контейнеров: 
```bash
docker compose down
```

### 2. Локальный запуск в терминале (Опционально)

Сервис может работать через in-memory хранилище для быстрой локальной разработки:
```bash
STORAGE=memory go run ./cmd/server/main.go
```

Если вы хотите запустить проект локально с использованием PostgreSQL, сначала поднимите базу данных:
```bash
docker compose up -d postgres
```
А затем запустите сервер:
```bash
STORAGE=postgres go run ./cmd/server/main.go
```

Переменные окружения (настраиваются через `.env` или флаги ОС):
- `STORAGE` - определяет тип используемого хранилища (`memory` или `postgres` по умолчанию)
- `SERVICE_PORT` - внутренний порт HTTP-сервера (по умолчанию: `8080`)
- `BASE_URL` - базовый URL, на котором будут хранится те самые короткие ссылки (по умолчанию: `http://localhost:8080`)
- `DB_HOST` - хост базы данных Postgres (по умолчанию: `localhost`)
- `DB_PORT` - порт базы данных Postgres (по умолчанию: `5432`)
- `DB_USER` - пользователь БД (по умолчанию: `postgres`)
- `DB_PASSWORD` - пароль БД (по умолчанию: `postgres`)
- `DB_NAME` - имя базы данных (по умолчанию: `urls`)
- `MIGRATE_FOLDER` - папка с миграциями, которые применятся при старте (по умолчанию: `./migrations`)

## Makefile команды

Для упрощения работы с сервисом добавлены следующие `make` команды:

### Запуск и остановка приложения

```bash
make docker-up       # сборка и фоновый запуск контейнеров (БД + сервер)

```

### Локальный запуск

```bash
make run-memory      # локальный запуск через in-memory хранилище

```

### Тестирование

```bash
make test            # прогон unit-тестов
```

## API Эндпоинты

### 1. Создать короткую ссылку
- `POST /api/v1/links/reduce`
- Тело запроса (JSON):
```json
{
  "url": "https://ozon.ru/test-link"
}
```
- Пример успешного ответа:
```json
{
  "url": "https://ozon.ru/test-link",
  "short_code": "qwerty123_",
  "short_url": "http://localhost:8080/qwerty123_"
}
```

### 2. Получение полной ссылки по короткой
- `GET /<short_code>` 
- Ищет в базе данных короткую ссылку по короткому коду и возвращает полную ссылку.
*(Короткий код, полученный в результате POST-запроса надо передавать напрямую в url)*.

## Ход решения

### 1. Единая точка управления конфигурацией
Для конфигурирования сервиса использована библиотека `caarlos0/env`. Чтобы упростить запуск из коробки, всем полям конфигурации заданы значения по умолчанию (`envDefault`).

### 2. Разделение хранилищ
ТЗ требовало возможность выбора типа хранилища через аргументы запуска. Логика реализована через общий контракт `storage.Repository`. На этапе инициализации `app.go`, сервис смотрит переменную `STORAGE` и прокидывает нужную реализацию бд.

### 3. Единый формат ошибок
Чтобы можно было получить понятный ответ от сервера, был разработан кастомный "перехватчик" `HTTPErrorHandler` (`apperrors`). Вместо того чтобы вручную собирать JSON-ошибки в каждом хендлере, слой бизнес-логики использует только доменные ошибки, а "перехватчик" сам преобразует их в коды и на выходе видно будет понятный json с ошибкой.

### 4. Алгоритм генерации коротких ссылок
Алгоритм генерации специально сделан максимально простым и понятным: сервис генерирует случайную последовательность символов длиной 10, потом проверяет, существует ли уже такая короткая ссылка в хранилище, и если код свободен — сохраняет его. Если произошла коллизия, генерирует новый код и повторяет попытку.

Дополнительно сервис гарантирует, что для одного и того же оригинального URL всегда возвращается одна и та же короткая ссылка: сначала выполняется поиск по original_url, и если ссылка уже существует, сервис возвращает ее, а не создает новую.
